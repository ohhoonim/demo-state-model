# Anemic vs Rich Domain Model

## Anemic domain model

객체는 단순히 데이터를 담아 전달하는 컨테이너(DTO 형태) 역할만 수행하며, 실제 비즈니스 로직은 Service나 Manager 등의 계층에서 절차적으로 처리합니다.

- **특징**:
    - 객체 내부에는 필드와 getter/setter만 존재하는 경우가 많습니다.
    - 행위가 데이터와 분리되어 있어 절차지향 프로그래밍과 유사한 형태를 띱니다.
    - 전통적인 Spring/JPA 환경에서 매우 흔하게 사용되는 방식입니다.
- **장점**: 구조가 단순하여 초기 개발 속도가 빠르고 이해하기 쉽습니다.
- **단점**: 비즈니스 로직이 Service 계층에 집중되면서 Service가 비대해지고, 데이터 객체의 캡슐화가 깨져 데이터 무결성을 보장하기 어렵습니다.

---

## Rich domain model

객체가 자기 자신의 데이터를 스스로 관리하고, 그 데이터와 관련된 비즈니스 규칙(유효성 검증, 상태 변경 등)을 직접 수행합니다.

- **특징**:
    - 데이터와 행위(메서드)가 한곳에 모여 있어 응집도가 높습니다.
    - 객체의 상태를 외부에서 함부로 변경할 수 없도록 캡슐화가 잘 유지됩니다.
    - 도메인 주도 설계(DDD)에서 지향하는 방식입니다.
- **장점**: 코드의 재사용성이 높아지고, 비즈니스 로직이 분산되는 것을 막아 유지보수에 유리합니다.
- **단점**: 객체가 무거워질 수 있으며, 여러 도메인이 복합적으로 얽힌 로직을 한 객체에 담기 어려울 수 있습니다.

---

## 비교

| 구분         | 풍부한 빈 (Rich)           | 빈약한 빈 (Anemic)       |
|------------|------------------------|----------------------|
| 중심 패러다임    | 객체지향 (Object-Oriented) | 절차지향 (Procedural)    |
| 비즈니스 로직 위치 | 도메인 객체 내부              | Service 계층           |
| 데이터와 행위    | 결합되어 있음                | 분리되어 있음              |
| 캡슐화 수준     | 높음 (상태 변경 제어 가능)       | 낮음 (외부에서 자유롭게 상태 변경) |
| 유지보수성      | 대규모 복잡한 시스템에 유리        | 단순하고 규모가 작은 시스템에 유리  |

---

## Anemic domain model 의 특징

**1. anemic 도메인 모델의 특징**

- 겉으로 보기에는 실제 도메인 모델처럼 보입니다. 도메인 영역의 명사들을 딴 객체들이 있고, 객체 간의 복잡한 관계도 설정되어 있습니다.
- 하지만 내부를 들여다보면 로직이 거의 없습니다. 단순히 데이터를 담는 가방(Bags of getters and setters) 역할만 수행합니다.
- 비즈니스 규칙, 유효성 검사, 계산 로직 등은 모두 `Service`나 `Manager`라고 불리는 클래스들로 옮겨져 있습니다.

**2. 왜 문제인가? (안티 패턴인 이유)**

- **객체지향의 실종:** 객체지향 설계의 기본은 데이터와 그 데이터를 사용하는 로직을 하나로 묶는 것입니다. 이를 분리하는 것은 사실상 절차지향 프로그래밍으로 회귀하는 것입니다.
- **비용과 편익의 불균형:** 도메인 모델을 구축하기 위해 ORM 설정 등 많은 비용을 지불하면서도, 정작 도메인 모델이 주는 강력한 이점(복잡성 관리, 캡슐화 등)은 전혀 누리지 못합니다.
- **낮은 응집도:** 로직이 도메인 객체가 아닌 여러 서비스에 흩어져 있어, 특정 비즈니스 규칙을 찾거나 수정하기가 어렵습니다.

**3. 발생 원인**

- 많은 개발자가 실제 객체지향 설계보다는 데이터베이스 중심의 사고방식에 익숙하기 때문입니다.
- 기술적인 레이어 구분(Layered Architecture)을 로직과 데이터를 분리하는 것으로 오해하는 경우가 많습니다.

**4. 대안: 풍부한 도메인 모델 (Rich Domain Model)**

- 로직을 서비스 계층에서 도메인 객체 내부로 옮겨야 합니다.
- 서비스 계층은 얇게(Thin) 유지하며, 도메인 객체들 사이의 협력을 조정하는 역할만 수행해야 합니다.
- 진정한 도메인 모델은 상태(State)뿐만 아니라 비즈니스 상황을 반영하는 행위(Behavior)를 가져야 합니다.
